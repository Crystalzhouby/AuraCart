# 多 Agent 导购功能实现方案

## 1. 功能点概要链路

### 1.1 多轮会话入口

用户通过对话接口发送：

```json
{
  "session_id": "xxx",
  "message": "用户输入",
  "selected_option": {
    "key": "A",
    "payload": {}
  }
}
```

后端读取会话记忆，进入导购编排器。

### 1.2 查询改写

链路：

```text
用户当前输入 + 会话记忆
  → 查询改写 Agent
  → 完整购物需求 + 实体约束
```

目标：

- 处理“再便宜点”“不要这个”“第二个呢”等上下文表达。
- 保留预算、品类、否定条件、人群、场景等信息。

### 1.3 意图识别

链路：

```text
改写结果 + 用户原始输入
  → 意图识别 Agent
  → 路由到不同流程
```

支持：

- 纯闲聊
- 潜在购物需求
- 明确单品推荐
- 场景方案
- 商品对比
- 购物车动作
- 兜底未知

### 1.4 需求探索

适用于用户还没有明确要买什么的情况。

```text
潜在痛点
  → 需求探索 Agent
  → 自然追问 + 快捷回复候选
```

示例：

```text
用户：最近皮肤状态好差
系统：是更偏熬夜暗沉、出油长痘，还是干燥起皮？
```

### 1.5 需求澄清

适用于用户已经知道大概要买什么，但条件不足。

```text
明确品类 + 已收集约束
  → 需求澄清 Agent
  → clarify 或 ready
```

信息不足时返回追问和快捷回复；信息足够时进入检索。

### 1.6 商品检索

链路：

```text
完整需求
  → 商品检索 Agent 生成检索计划
  → 检索工具执行关键词/向量/结构化过滤
  → 候选商品重排
```

检索工具应复用或改造当前 `QueryParser`、`Retriever`、`Merger`、SKU 补全能力。

### 1.7 推荐生成

链路：

```text
候选商品 + 用户需求 + RAG 知识块
  → 推荐生成 Agent
  → 回复文本 + 推荐商品理由
```

推荐生成必须接入：

- 商品标题
- 品牌
- 类目
- SKU 价格
- FAQ
- 用户评价
- 营销描述

### 1.8 快捷回复

每个业务分支末尾统一调用快捷回复 Agent。

```text
当前状态 + 系统回复 + 商品结果
  → 快捷回复 Agent
  → A/B/C 选项
```

快捷回复用于降低输入成本，不用于强行引导购物。

## 2. 主要优点

- 业务链路清晰，符合真实导购从闲聊、探索、澄清到推荐的过程。
- 各 Agent 职责单一，便于单独测试和迭代 Prompt。
- 能复用当前 `ymf_backend` 已有检索链路，避免推翻重写。
- 快捷回复提升用户体验，也能降低后端理解难度。
- 会话记忆能支持多轮追问、反选、对比和加购。
- 商品事实由工具层读取，能降低大模型幻觉风险。

## 3. 主要风险

- 多 Agent 串行调用可能导致延迟增加。
- LLM JSON 输出可能不稳定，需要解析和降级策略。
- 商品数据字段不完全一致，可能影响检索和推荐理由。
- 快捷回复如果生成过度营销，会破坏聊天体验。
- 需求探索和需求澄清边界容易混淆。
- 当前后端接口偏搜索，需要新增对话编排入口。

## 4. 实现复杂度

| 功能 | 复杂度 | 说明 |
| --- | --- | --- |
| 会话记忆 MVP | 中 | 内存版简单，持久化后复杂 |
| 查询改写 Agent | 中 | 需要稳定 JSON 和上下文合并 |
| 意图识别 Agent | 中 | 分类清晰，但要处理模糊输入 |
| 需求探索 Agent | 中 | 容易过度主动，需要约束 |
| 需求澄清 Agent | 中 | 需要类目槽位配置 |
| 商品检索 Agent | 中高 | 要和现有检索链路对齐 |
| 推荐生成 Agent | 中 | 需要接入 RAG 知识块 |
| 快捷回复 Agent | 低中 | 输出简单，但体验要求高 |
| LangGraph 接入 | 中高 | 建议流程稳定后再接 |

## 5. 可测试性

每个 Agent 都应独立测试：

- 输入固定 JSON，断言输出可解析。
- 断言否定条件不丢失。
- 断言纯闲聊不会强行导购。
- 断言快捷回复最多 3 个。
- 断言推荐结果不包含商品库外事实。

建议准备测试用例：

- “推荐个手机”
- “最近皮肤状态好差”
- “推荐 200 元以内适合油皮的防晒”
- “不要日系，也不要含酒精”
- “第二个和第三个哪个好”
- “把第一款加购物车”
- “你好呀”

## 6. 可交付性

建议分阶段交付。

第一阶段：

- 手写 Python 导购编排器。
- 内存会话记忆。
- 查询改写、意图识别、需求探索、需求澄清、快捷回复。
- 接入现有检索链路。

第二阶段：

- 推荐生成接入 FAQ、评价、营销描述。
- 场景方案和商品对比。
- 购物车动作解析。

第三阶段：

- 持久化会话记忆。
- LangGraph 编排。
- 更完整的性能优化和链路观测。

## 7. 推荐技术路线

当前阶段推荐：

```text
FastAPI + Python 手写编排器 + Pydantic Schema + 现有 LLMService + 现有检索链路
```

流程稳定后再考虑：

```text
LangGraph 编排多 Agent 状态机
```

暂不建议直接引入 DeerFlow 2.0 作为核心依赖。
