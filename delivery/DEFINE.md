# 多 Agent 导购功能问题定义

## 1. 最终交付物

- 一套基于当前 `ymf_backend` 分支可继续编码的多 Agent 导购方案。
- 支持多轮对话中的查询改写、意图识别、需求探索、需求澄清、商品检索、推荐生成和快捷回复。
- 新增或改造后端对话接口，支持 `session_id`、用户自由输入、快捷回复选项、商品推荐结果。
- 形成可演示链路：用户输入 → Agent 判断 → 追问或检索 → 推荐回复 → ABC 快捷回复。
- 文档交付：
  - `delivery/MULTI_AGENT_DESIGN.md`
  - `delivery/DEFINE.md`
  - `delivery/PLAN.md`
  - `delivery/CON_PLAN.md`

## 2. 硬约束

- 必须基于本仓库现有 `ymf_backend` 分支继续推进。
- 商品事实必须来自项目数据和数据库字段，不能由模型编造。
- 价格、SKU、商品标题、品牌、类目、FAQ、评价等必须可回溯到商品库。
- 用户可以点 ABC 快捷回复，也必须保留自由输入。
- 快捷回复只用于降低输入成本，不用于强行引导用户购物。
- Agent 输出应尽量使用可解析 JSON，避免纯自然语言中间结果。
- 检索、购物车、数据库操作属于确定性工具层，不应完全交给大模型自由执行。

## 3. 隐含要求

- 系统需要会话记忆，否则无法处理“再便宜点”“第二个”“不要这个”等上下文表达。
- 需要区分纯闲聊、潜在购物需求、明确购物需求、场景方案、商品对比、购物车动作。
- 需求不明确时应先追问，不应直接推荐大量无关商品。
- 用户只是聊天时，也可以给快捷回复，但选项应自然延续对话。
- 推荐回复要能解释“为什么适合用户需求”，而不是只列商品。
- 后端接口要方便 Android 客户端渲染文本、商品卡片和快捷回复。

## 4. 任务完成边界

本阶段完成到以下程度即可认为任务闭环：

- 多 Agent 职责边界明确。
- 每个 Agent 的输入输出和 Prompt 方向明确。
- 后端模块拆分、接口结构、核心数据结构明确。
- 能指导后续编码实现，不要求本阶段完成全部代码。
- MVP 范围明确：查询改写、意图识别、需求探索、需求澄清、商品检索、推荐生成、快捷回复、会话记忆。

本阶段不要求：

- 完成多模态拍照找货。
- 完成真实下单支付。
- 完成完整 DeerFlow 2.0 接入。
- 完成生产级用户画像长期记忆。

## 5. 不确定项

- 是否最终采用 LangGraph 编排，还是先用 Python 手写编排器。
- 会话记忆 MVP 使用内存、PostgreSQL 还是 Redis。
- Android 端快捷回复 UI 的最终样式和事件协议。
- 商品库存字段是否全部可靠；部分原始数据可能只有价格和 SKU，没有真实库存。
- 当前数据库同步和导入链路是否需要先修复后再接 Agent。

## 6. 风险点

- Agent 过多导致首轮延迟变高。
- 多 Agent JSON 输出不稳定，导致解析失败。
- 查询改写丢失否定条件，例如“不含酒精”“不要日系”。
- 需求探索过度主动，导致用户只是闲聊却被强行导购。
- 快捷回复选项生成不自然或太像营销话术。
- 推荐生成未接入 FAQ、评价等 RAG 知识，推荐理由不够可信。
- 当前 `QueryParser` 只适合检索拆解，不能直接承担多轮改写和澄清。
- 如果一次性引入 LangGraph、向量库、会话库和多 Agent，交付复杂度会过高。
