# /search/stream中的查询优化

## 阶段1查询分解优化

拆解出的semantic类型查询须是评价类短句，如“保湿效果好”，“充电慢”等，而不是“防晒效果”这种名词；
拆解出的keyword类型查询须是用户查询中比较独特具体的关键词，如“适合学生用的 iPhone，要拍照清晰”，应拆解为[iPhone, 学生]
拆解structured_filter条件时，应参考product和sku表中的字段，重点把用户查询中与这些字段有关的属性提取出来，此外需要注意sku表中的properties字段，该字段包含多种属性，不同sku对象的properties字段属性还存在差异。

## 阶段2多策略检索优化

首先需要明确一下最终的检索结果单位是销售单品sku。
我觉得阶段1把用户查询分解为多个带有独立意图的子查询后，子查询之间的关系不应当是做"并集"，而应该是做"交集"，以”推荐一款200元以下的防晒霜“为例，拆解出来的200元以下的用户意图应当是其他的keyword和semantic查询的硬约束（用户需要防晒效果好的，即使某个产品语音再怎么匹配但是价格大于200，那也不应该被检索出来）。
为了到达这个目的，我认为应该将多个带有独立意图的子查询在检索时，子查询的协作关系是这样的：
- structured_filter类型查询不单独做检索，作为其他类型检索的硬性约束条件；
- keyword和semantic类型检索将structured_filter子查询给出的条件作为硬性约束条件，避免筛选出不满足用户硬性需求的商品；
- 应该将不同的semantic类型检索子句作为不同的相似度查询条件合并起来做检索，多条 semantic 子查询需独立进行相似度计算，将各自得分融合为综合分数（当前融合方式先采用直接加和），最终根据综合分数从高到低排序，截取 Top K 作为结果；
- 对于keyword检索，应当直接使用BM25做检索，得到Top K结果。
- keyword和semantic类型检索是双路并行的，最终通过RRF综合两类查询得到最终的综合得分最高的Top K结果。
### 示例1
用户查询：推荐一款200元以下、防晒效果好、质地清爽的防晒霜
1. structured_filter：品类=防晒霜，价格 ≤ 200元
2. keyword：["防晒霜"]
3. semantic：
   - 子查询1：产品防晒效果是否出色
   - 子查询2：产品质地是否清爽

### 检索执行流程：
1. 先提取出 price≤200 AND category = "美妆护肤" 的structured_filter硬性条件；
2. 关键词检索：基于提取出的keyword，检索出满足structured_filter硬性条件且BM25得分前K的sku；
3. 语义检索：基于提取出的两条semantic子查询，分别计算两条语义子查询与product_review中数据的相似度得分，并将两个子查询的语义得分做加和，作为语义综合得分，进一步按总分排序，返回得分前K的sku；
4. 通过RRF，融合关键词检索结果和语义检索结果得到综合排名结果，返回Top K的sku。
