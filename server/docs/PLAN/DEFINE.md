# 问题定义

## 1. 最终交付物

一个基于 RAG 的智能导购后端 API 系统，具备以下能力：
- 接收用户自然语言商品购买意图，返回匹配商品及推荐理由
- 支持"模糊推荐"和"条件筛选"两类场景
- 提供流式 API（WebSocket 或 SSE）
- 支持混合检索（向量语义检索 + 关键词精确匹配）

## 2. 硬约束

| 约束项 | 具体要求 |
| :--- | :--- |
| 后端框架 | FastAPI |
| 大模型 | Doubao-Seed-2.0-lite |
| Embedding 模型 | 豆包/千问等平台（需能表达用户购买需求与商品信息） |
| 数据库 | 必须同时支持向量检索和关键词检索（Milvus/Qdrant/Chroma/pgvector 等任选） |
| 流式通信 | 必须支持 WebSocket 或 SSE 长连接 |
| 反幻觉 | 严禁 AI 编造不存在的优惠券、功能或商品信息 |
| 数据一致性 | 推荐结果中的价格、库存等关键参数必须准确且实时生效 |
| 数据更新 | 商品信息（描述、价格、库存）需支持实时更新 |

## 3. 隐含要求

- **意图分类**：需区分"随便看看"（模糊浏览）与"明确购买倾向"（精准检索），触发不同检索策略
- **Chunking 策略自设计**：规格未给定具体方案，需自行设计商品信息的切分粒度，平衡召回率与精确度
- **Prompt Engineering**：需设计系统提示词控制输出格式与风格
- **工程健壮性**：需处理并发、超时、降级等场景
- **可扩展网关**：向量检索与大模型调用的接口网关需预留扩展能力
- **结构化数据与向量数据协同**：价格、库存等精确字段存储在 products 关系表中，需与向量检索结果关联，保证推荐结果中价格/库存的实时准确性

## 4. 任务完成边界

**在范围内：**
- 商品知识库搭建（JSON 数据导入 → Chunking → 向量化 → 索引构建）
- 结构化 products 表设计（存储价格、库存等实时可变字段）
- 混合检索引擎（向量检索 + 关键词检索 + 结构化字段过滤）
- 意图识别与检索策略路由
- RAG 链路（检索结果 → Prompt 组装 → 大模型生成推荐理由）
- 流式 API 端点
- 数据实时更新接口（products 表的价格/库存同步）
- 基础的服务稳定性保障（超时、降级）

**明确在范围外：**
- 多轮对话能力（已明确暂不交付）
- 图片 embedding / 多模态检索（暂不需要）
- 前端/UI 界面
- 用户认证/权限系统
- 部署与运维配置（本地运行即可）

**数据集规模：** 4 个品类（美妆护肤、数码电子、服饰运动、食品生活），约 100 个商品（每品类约 25 个），每个商品含 JSON 结构化数据 + 商品图片。

## 5. 风险点

| 风险 | 说明 |
| :--- | :--- |
| **结构化与向量数据一致性** | products 表（价格/库存）与向量索引（商品描述）是两套存储，商品新增或下架时需保证两边同步，否则可能出现"搜到但价格不对"或"搜到已下架商品" |
| **Chunking 策略无参考** | 商品信息包含营销文案、FAQ、用户评价等多种文本类型，切分方式直接影响检索质量，需多轮实验 |
| **混合检索权重调优** | 向量检索与关键词检索的融合策略（权重分配、排序合并）无明确标准，需根据实际效果迭代 |
| **大模型幻觉控制** | 即使有 RAG 检索结果约束，大模型仍可能产生幻觉（如编造折扣、功能），Prompt Engineering 效果不确定 |
| **Embedding 模型选型** | 豆包/千问 embedding 对电商领域商品的语义表达能力未经验证，可能存在领域适配问题 |
| **数据集规模有限** | 约 100 个商品，4 个品类。小数据集下向量检索的召回效果可能不明显，关键词检索可能更实用 |
