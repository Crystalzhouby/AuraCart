# 商品数据明细

## 商品基础信息
当导入一个json格式的商品信息时，商品基本信息包含以下字段
- product_id 商品唯一编号
- title 标题
- brand 品牌
- category 品类
- sub_category 子品类
- base_price 基础价格
- image_path 参考图片地址
- skus列表 多种销售单品

具体来说对于单个skus，包含以下字段
- sku_id 销售单品唯一编号
- properties 商品属性，不同销售单品商品不同，例如：
    - 美妆护肤类为容量；
    - 数码电子类为存储空间、颜色、版本；
    - 服饰运动类为尺码、颜色；
    - 食品饮料为口味、数量、包装、单盒容量
    甚至对于同一品类或子品类的商品也不同，例如：
康师傅的属性为口味、规格、包装，同为方便食品子类的日清的属性为容量、数量、口味；
- price 同一product_id下的不同销售单品价格是不同的。

## 商品评价信息
商品信息中还包含商品的评价信息，即rag_knowledge，包含以下字段：
- marketing_description 商品营销描述
- official_faq列表 官方常见问题解答
    - question 针对商品的疑问
    - answer 官方回复
- user_reviews列表 用户评论
    - nickname 作出评论的用户的昵称
    - rating 商品评分
    - content 评论内容

# 检索数据设计方案

## 用于关键词检索的数据

用两个数据表存储，分别是product和skus。
- product中存储了除了skus的所有商品基础信息，该表的意义是存储某一商品的基础信息。
- skus中存储了所有的销售单品信息，除了包含了所有的skus字段，还包含一个外键product_id，说明其所属的商品，还包含库存stack等信息，后续可能还会补充enable(是否上架)等信息。
此外，以上两表还包含创建时间和更新时间

## 用于语义检索的数据

用向量数据库中的一个Collection，product_review存储，包含以下字段：
- content 评论信息的内容
- dense_vector embedding模型生成的该内容的稠密向量
- metadata 评论元信息
    - product_id 评论所属的商品编号
    - source 评论所属的来源类别marketing_description/official_faq/user_reviews
    - create_time 商品信息创建时间
    - update_time 商品信息更新时间

其中评价信息不仅仅包含rag_knowledge中的内容，还包含商品基础信息（基础信息也需要支持模糊查询，例如品牌需要不是日系的）


# 检索设计

对于检索，我的认识如下：
## 检索句层面

核心问题：
    用户的查询用于往往是带有多种意图的，为了精确地理解用户的查询意图，需要把用户的查询按照语义拆分为多个意图较为单一的查询句或关键词；
    - 例子1：“推荐防晒霜，但我不要含酒精的，也不要日系品牌”，对于该查询，我认为至少包含三个意图，（1）用户需要防晒霜，（2）防晒霜的配方不包含酒精、（3）品牌不是日系的。
    - 例子2："200 元以下的蓝牙耳机有哪些？"对于该查询，我认为至少包含三个意图，（1）用户需要耳机；（2）价格低于200；（3）耳机需要是蓝牙的。
难点：
    - 如何将带有复杂意图的用户查询拆分成多个单一意图的用户查询；
    - 对于每个单一意图的用户查询，有的需要精确匹配（价格、尺码、品牌等）关键词精确匹配；有的需要按照模糊语义匹配（如适合油皮的洗面奶、防晒效果好）语义向量相似度比较；有的关键词也需要模糊匹配（例如品牌需要是日系的），用品牌关键词模糊匹配“日系”；
    - 不同商品在不同查询子句下有不同得分，如何综合这些得分，获得综合排名；

## 检索数据库层面

核心问题：
    1. 商品评价信息复杂，尤其是用户评论，其表达意图也是较为复杂的，需要做拆分，此外同一商品下的不同评论或官方faq表达的意图可能是矛盾或重复的，此时如何合并这些信息。
    2. 用户的一些查询意图较为宽泛模糊，不仅仅是针对较为主观的评论，还会对一些客观的基础信息做模糊查询，如品牌需要不是日系的；那么为了保障存储效率和检索效率，检索策略需要怎么来设置。
    3. 商品信息需要支持实时更新；
    4. 如何将结构化的商品信息转换为自然语言描述的查询句，进一步转换为向量，例如user_reviews包含评论用户名+评分+评论内容，如何转换为一个自然语句？
    4. 为支撑上述检索能力，需要商品信息的存储方案怎么设计，目前较为简单的设计，后续还需要修改：
        - 商品基础信息存储到关系型数据库的表中，即product和skus；
        - 商品评价信息存储到向量数据库中用于模糊查询；
