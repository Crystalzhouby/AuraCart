# DEFINE.md — 多轮对话自然语言价格调整 需求分析

> 输入：`SPEC.md`（同目录）

## 1. 功能需求

- **FR1 自然语言价格下调**：当用户在当前轮次用自然语言表达"更便宜/更平价/预算有限"等意图时，系统能识别并相应降低 `max_price`（和/或 `min_price`），而非机械维持上轮的数值价格。
- **FR2 自然语言价格上调**：对称支持"可以贵一点/档次高一点"等上调意图，调整 `min_price` 或 `max_price`。
- **FR3 调整幅度与语言强度挂钩**：LLM 根据用户用词的强烈程度（"稍微便宜" vs "越便宜越好"）动态决定调整比例，而非固定比例。
- **FR4 兼容现有显式数值变更**：不改动已有的"后说数字覆盖前说数字"的冲突处理逻辑。
- **FR5 不影响非价格场景**：对于不涉及价格调整的对话轮次，行为与当前保持一致。

## 2. 性能需求

- **PR1 零额外 LLM 调用**：仅修改现有 Step 3 prompt，不增加网络请求。
- **PR2 延迟增加 < 5%**：prompt 改动仅增加少量 token，不应带来可感知的延迟增长。

## 3. 最终交付物

1. 修改后的 `extraction_prompt.py`（`EXTRACTION_STEP3_SYSTEM`）
2. 对应的测试用例（`tests/test_extraction.py`）
3. 验证通过的测试结果（以 SPEC.md 中的例子：第一轮"300元以下的防晒霜" → 第二轮"请推荐更平价的产品"）

## 4. 硬约束

- **C1** 只改 prompt，不改 extraction 节点的 Python 代码流程
- **C2** 不改数据库 schema
- **C3** 不改 API 接口
- **C4** 不能破坏现有显式数值提取的准确性

## 5. 隐含要求

- **IR1** 调整后的价格应保持业务合理性（如 `min_price >= 0`，`min_price < max_price`）
- **IR2** 提示词应包含中文示例，覆盖至少"降价"和"提价"两类场景
- **IR3** 如果上下文无显式数值价格（如首轮就是"推荐便宜的"），直接按常规语义提取，不做相对调整

## 6. 任务完成边界

| 范围内 | 范围外 |
|--------|--------|
| 修改 `EXTRACTION_STEP3_SYSTEM` prompt | 修改 scenario_gen 路径的价格提取 |
| 自然语言对已有数值价格的**相对调整** | 从纯自然语言中提取**绝对价格**（"平价"→具体数字） |
| min_price / max_price 的上调和下调 | 品牌、品类等其他约束的自然语言调整 |
| 通过 SPEC.md 示例验证效果 | 全面的多场景回归测试 |

## 7. 风险点

- **R1 LLM 不稳定**：纯 prompt 方案依赖 LLM 推理能力，同一输入可能产生不同调整幅度。缓解：prompt 中给出明确的调整逻辑框架，减少自由度。
- **R2 过度调整**：连续多轮"再便宜点"可能导致价格收敛到 0。缓解：prompt 中明确调整底线（min_price 不低于 0，max_price 下调有下限）。
- **R3 误触发**：用户说"这个不错，便宜好用"（评价而非要求）可能被误判为调整意图。缓解：prompt 中区分"评价性语言"和"指令性语言"。
